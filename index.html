First Software Documentation Generation Challenge

The Software Documentation Generation Challenge aims to spur research to increase automation of software documentation creation and maintenance.

ABOUT

Documentation is an essential component of the software development process, yet efficiently writing and maintaining high-quality documentation is extremely challenging. Automatically-generated On-Demand Developer Documentation (OD3) has the potential to make information on software more accessible to developers and cheaper to produce and maintain.

The Software Documentation Generation Challenge aims to spur research to increase automation of software documentation creation and maintenance. Participation in the challenge involves developing a prototype tool and operating the tool as part of a live competition. The first DocGen Challenge Competition will be co-located with the IEEE International Conference on Software Maintenance and Evolution (ICSME 2018). The DocGen-1 Competition will feature a live, juried tool evaluation, presentations of the best approaches, and a poster session. The extended summaries describing the entries will be published in the workshop proceedings. The best entries will be recognized with awards.

2018 CHALLENGE

The challenge is to build an automated system that can create, on-demand, reference documentation for a Java class. The documentation should address, as best as possible, the information needs of an experienced programmer who is considering using the class as part of a software development task. The documentation will provide information helpful to understand the design of the class and how to use it. Examples of the types of information that the document could integrate include: key methods, usage constraints, usage examples, the role of the class in applicable design patterns, known issues and limitations, etc.

SUBMISSION

An entry to the OD3 Challenge is a program that takes as input the fully qualified name of a class from the Apache POI project and generates an output that contains the reference documentation for the input class in html format.

The target system for this challenge is the Apache POI project, version 3.17-final (r31807825, commit 219dff00e61e9ad73a16b73a430cb80954f6262c, Sept 8, 2017). The source code of the project can be found on Github.

The document generator can rely only on the following resources:

  • Source code. The generator can use existing comments, but these will be considered a baseline when evaluating solutions.
  • Version control data
  • Mailing list
  • Issue tracking data
  • Stack Overflow threads for POI

We also provide a preprocessed dataset of the above information, which includes the following:

  • A set of all the commits of the projects.
  • A set of all the issues of the project, including links to the commits that address the issue, if any
  • A set of all the messages sent to the mailing lists
  • A set of UML diagrams that visually represent a selected group of relations between program elements.

Contestants are free to use these preprocessed data sets, develop their own tools to process the original information found in the repositories, or use a combination of both. The solution can involve human-produced intermediate data and annotations: however, any manual input to the generation process must be clearly detailed. Entries that focus on one aspect of the reference documentation are explicitly encouraged, in particular if they are novel and innovative

Interested teams of participants can enter the challenge in one of two categories: competition, or exhibition. The Competition category will only accept entries that address the specific DocGen-1 Challenge. The Exhibition category will accept any demonstration of documentation generation technology, for any programming language, technology, or types of documents. Entries in the Exhibition category will be invited to present at the competition and participate in the poster session, but will not be evaluated as part of the competition. It is expected that the specification of the following challenge will be influenced by strong entries in the exhibition categories.

Participation in the DocGen-1 Challenge involves two phases: the qualification and the competition. All participants in both categories must qualify. All qualifying entries will be invited to the DocGen-1 event. Only entries submitted to the competition category will be evaluated as part of the challenge competition.

Qualification

To qualify for participation in the challenge, a team of contestants must submit of a 2-page description of the key idea(s) used to generate the reference documentation. The description should contain the following information:

  • Authors
  • Data sources used
  • Brief overview of technique(s) used
  • Preliminary results, including an HTML link to the generated reference documentation for class ?????

All submissions must be formatted according to the ICSME Formatting Instructions and must be submitted through EasyChair.

Competition

On the day of the competition, qualified entries will be evaluated live. A list of target class names will be provided to the competitors, who will use their tools to generate the respective reference documentation. For each input class, the reference documentation must be generated in less than 2m18s. The generated documents will be evaluated live by a panel of judges and by the workshop attendants based on:

  • Correctness of the content
  • Usefulness of the content in using the class
  • Conciseness of the document
  • Presentation of the information
